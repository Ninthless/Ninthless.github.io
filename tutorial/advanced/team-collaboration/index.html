团队协作规范

把个人 Vibe Coding 实践升级为团队标准，让整个团队都能稳定交付。

团队规范体系

graph TD
    A[个人实践] --> B[团队规范]
    B --> C[项目规则文件]
    B --> D[提示词模板库]
    B --> E[验收标准清单]
    B --> F[复盘机制]

项目规则文件

每个项目维护一个 .trae/rules/project_rules.md，包含：

# [项目名] Vibe Coding 规则

## 技术栈
[列出所有主要依赖和版本]

## 目录结构约定
[说明各目录的职责和边界]

## 代码约定
[命名规范、文件组织、注释要求]

## 禁止事项
[明确列出不允许 AI 做的事情]

## 验收标准
[每次任务完成后必须运行的命令]

## 常见陷阱
[本项目中 AI 容易犯的错误]

提示词模板库管理

模板分级

级别 说明 维护方式
通用模板 跨项目可用 团队 Wiki
项目模板 项目特定约定 项目仓库
个人模板 个人习惯 个人笔记

模板评审流程

  1. 开发者提交新模板到团队 Wiki
  2. 至少 2 人在实际任务中验证
  3. 记录验证结果和适用场景
  4. 正式纳入团队模板库

Code Review 规范

AI 生成代码的额外检查项

在常规 Code Review 基础上，增加：

  • 代码是否与项目架构约定一致
  • 是否引入了不在 package.json 中的依赖
  • 错误处理是否完整（AI 容易遗漏边界情况）
  • 安全敏感逻辑是否经过人工深度审查
  • 测试覆盖率是否满足要求

PR 描述模板

## 变更摘要
[简述做了什么]

## AI 协作说明
- 使用的提示词模板：[模板名称]
- AI 生成的部分：[列出 AI 生成的文件/函数]
- 人工修改的部分：[列出人工调整的内容]

## 验证结果
- [ ] 单元测试通过：`[测试命令]`
- [ ] 类型检查通过：`[类型检查命令]`
- [ ] 构建成功：`[构建命令]`
- [ ] 手动验收：[验收步骤]

## 残余风险
[列出未覆盖的边界情况或已知限制]

团队复盘机制

每周 Vibe Coding 分享（15 分钟）

  • 本周最有效的提示词模式
  • 本周遇到的 AI 失败案例和解决方法
  • 新发现的工具或技巧

月度模板库更新

  • 收集本月的有效模板
  • 淘汰效果不好的模板
  • 更新项目规则文件

新人入职指南

新成员加入团队后的 Vibe Coding 上手路径：

  1. 阅读项目规则文件（.trae/rules/project_rules.md）
  2. 完成 3 个“带师傅”任务（有经验成员 Review）
  3. 独立完成 1 个小功能，提交 PR
  4. 参加第一次团队复盘分享

提示 · 关键原则

团队规范不是限制，而是让每个人都能快速上手的加速器。规范应该随着团队实践不断演进，而不是一成不变。
